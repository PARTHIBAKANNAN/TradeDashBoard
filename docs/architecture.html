01 — Topology

System topology & data plane

Three isolated tiers. The browser never touches the broker: all secrets and the raw broker websocket live only on the backend, which throttles the firehose of ticks into a single 1-second payload. A fourth tier — Supabase — handles who's allowed to open the dashboard at all, independent of the FYERS broker connection.

02 — Stack

Tech stack

Deliberately small: no ORM, no database, no message queue. Market state lives entirely in one Python process's memory; every dependency below earns its place.

🐍Backend

FastAPI 0.115 · Uvicorn (ASGI server) · APScheduler (IST-timezone cron jobs) · fyers-apiv3 (broker REST + websocket SDK) · PyJWT + cryptography (ES256 Supabase JWT verification) · pyotp (TOTP codes) · itsdangerous (signed session cookies via Starlette's SessionMiddleware) · python-dotenv · pytz.

⚛️Frontend

React 18 · Vite 6 (dev server + build) · Tailwind CSS 3 (semantic CSS-variable tokens for light/dark theming) · d3-hierarchy (squarified treemap layout for the Heatmap) · @supabase/supabase-js (client-side auth) · native EventSource (SSE) and Canvas 2D (range bars) — no charting library.

🔑Auth

Supabase Auth gates the dashboard itself (real user accounts, email/password). FYERS OAuth v3 (+ a TOTP-driven headless variant) gates the live market-data connection. The two are independent layers.

☁️Infra

Oracle Cloud VM (Ubuntu, Mumbai region) · Caddy (reverse proxy + automatic HTTPS via Let's Encrypt) · systemd (backend process supervision, auto-restart) · DuckDNS (free domain) · GitHub Actions (CI/CD, SSH deploy on every push to main).

03 — Boundary

Security model

The architectural boundary is the whole point of the BFF: credentials and the broker connection are quarantined server-side, and no shared secret is needed to verify the dashboard's own login either.

✕Never on the frontend

FYERS Client ID · Secret Key · TOTP seed · trading PIN · access token · the raw broker websocket · any Supabase JWT secret (there isn't one — see below).

✓Backend-only

Loaded from .env via python-dotenv; the daily FYERS token is cached to .token_cache.json (git-ignored, atomic write).

🛡️

Corporate TLS. On a network with SSL inspection, Python's bundled CA store rejects the broker's cert. pip-system-certs routes TLS through the Windows certificate store (which holds the corporate root CA), fixing requests, the FYERS SDK, and the websocket at once. This is a Windows-dev-only dependency — the Linux VM deploy skips it entirely.

🔐

No shared JWT secret. The backend verifies Supabase session tokens against the project's public JWKS endpoint (/auth/v1/.well-known/jwks.json, ES256 asymmetric keys) via app/supabase_auth.py — there is nothing sensitive to leak even if that verification code were exposed.

04 — Authentication

Login: two independent layers

"Logging in" actually means two separate things here, and mixing them up is the most common source of confusion. Layer 1 decides who may open the dashboard at all. Layer 2 decides whether the dashboard has a live connection to your FYERS trading account. A user can be logged into Layer 1 with FYERS completely disconnected (Layer 2 shows an amber "Connect FYERS" banner) — they're unrelated.

Layer 1 — Dashboard login (Supabase Auth)

1

Frontend authenticates against Supabase directly

The Login form (frontend/src/App.jsx) calls supabase.auth.signInWithPassword({ email, password }) using @supabase/supabase-js, initialized in frontend/src/lib/supabaseClient.js with the project URL + public (publishable) anon key — safe to ship to the browser by design.

2

Session JWT is exchanged for our own cookie

On success, the frontend POSTs { access_token } to /api/auth/login. The backend verifies that JWT's signature via Supabase's JWKS endpoint (no secret needed) and, if valid, sets request.session["user"] — a signed cookie (Starlette SessionMiddleware + itsdangerous).

3

Why a cookie, not just the JWT?

The live price stream (/api/stream) uses the browser's native EventSource, which cannot send custom headers — so an Authorization: Bearer token can't reach it. Bridging Supabase's identity check into a same-origin session cookie means every route, including SSE, stays gated with zero special-casing.

4

Every request re-checks the cookie

security.is_authenticated(request) gates every non-public route via the require_login FastAPI dependency. The gate is skipped entirely (dev mode) when SUPABASE_URL is unset.

⚠️

No public sign-up. The app only ever calls signInWithPassword — there is no sign-up form. New users are created manually in the Supabase dashboard (Authentication → Users → Add user), and "Allow new users to sign up" should stay disabled there.

Layer 2 — FYERS broker connection

Fyers access tokens expire every 24 hours. Rather than a manual OAuth redirect every day, the backend drives Fyers' programmatic login using your own credentials + a TOTP seed — the standard headless algo-trading pattern. A newly created FYERS app (or one whose permissions just changed) requires a one-time browser consent; every day after runs unattended.

The daily automated chain

1

Send login OTP challenge

POST /vagator/v2/send_login_otp_v2 — the base64 login id opens a TOTP session.

# api-t2.fyers.in
{ "fy_id": base64(FY_ID), "app_id": "2" }  → request_key  (sub: totp_login)
2

Verify TOTP

POST /vagator/v2/verify_otp — code from pyotp.TOTP(seed).now(). Generated boundary-safe (avoids the tail of the 30s window) with a one-shot retry.

{ "request_key": …, "otp": TOTP() }  → request_key  (sub: verify_pin)
3

Verify PIN → vagator token

POST /vagator/v2/verify_pin_v2 — the _v2 variant is required; the base64 PIN yields a short-lived login access token.

{ "request_key": …, "identity_type": "pin", "identifier": base64(PIN) }
    → data.access_token  (the "vagator" bearer)
4

Obtain the OAuth auth code

POST /api/v3/token with the vagator bearer. For an authorized app this returns a redirect URL carrying the real auth_code (sub: auth_code).

# api-t1.fyers.in
Bearer <vagator_token>
{ "fyers_id", "app_id", "redirect_uri", "appType",
  "response_type": "code", "create_cookie": true }
    → Url?auth_code=…&state=dashboard
5

Exchange for the API access token

SessionModel.generate_token() POSTs to /validate-authcode with appIdHash = SHA256("client_id:secret") + the auth code.

session.set_token(auth_code)
token = session.generate_token()["access_token"]  # valid until end of day
6

Cache atomically & use

Token is date-stamped and written to a temp file then os.replace()'d into .token_cache.json — atomic, so a mid-write process kill can never corrupt it. Powers the REST FyersModel and the FyersDataSocket feed (CLIENT_ID:token).

One-time browser consent

A freshly created (or newly re-permissioned) app isn't authorized against your account yet, so step 4 returns a pre-consent data.auth token with no real auth_code — which /validate-authcode rejects (-437 invalid auth code). Running manual_auth.py once (prints the FYERS authorize URL → open in a browser → log in → Allow → paste the redirected URL back into the terminal) exchanges a real, consented auth_code and caches the resulting token. After that, the chain above runs unattended — until FYERS-side permissions change again (see the gotchas table below).

In-app "Connect FYERS" (alternative to the CLI script)

The dashboard also exposes a browser-based path: an amber banner + Connect FYERS button (visible whenever fyers_connected: false) opens auth.build_login_url() in a new tab. FYERS redirects back to GET /callback?auth_code=…, which the backend exchanges and caches the same way — no SSH access required. Caddy explicitly proxies the bare /callback path (outside /api/*) to the backend for this to reach FastAPI instead of the SPA's index.html fallback.

05 — Data plane

Complete data-retrieval flow

Two ingestion paths feed one thread-safe in-memory state: a REST backfill at startup so no field is ever blank, and the live websocket for streaming ticks. A per-second SSE generator packages state for the client.

A · Startup REST backfill (DataEngine.backfill())

1Previous day

Daily candles (resolution:"D", ~12d) → last completed day's high / low / close for the benchmark & every stock.

2Today's ORB

30-min candles for today → C1–C4 high/low keyed by candle start-time, plus today's running high/low.

3Breakout-quality seed

5-min candles for 9:15–9:45 → candle-1 high/low reference + whether the range is two-sided (see Calculations).

4Quote seed

Batched quotes() (≤50/req) → an initial LTP so rows render fully before the first tick.

⏱Weekend / no-data guard

Steps 2–3 skip themselves entirely on Sat/Sun (no session data could possibly exist) instead of wasting the rate-limit budget on 170 guaranteed failures.

⛔Non-retryable abort

A permission-level error (e.g. -403) is app-wide, not per-symbol — the whole history pass aborts on the first occurrence rather than repeating the failure 170×.

06 — Engines

Mathematical calculations

Every tick recomputes these deterministic engines in memory before the next broadcast. All are pure functions, unit-tested in backend/tests/test_calculations.py.

Engine A

Intraday Relative Strength vs NIFTY 50

IRSi,t = Δ% Stocki,t − Δ% Indext

Each side is % change vs its own previous close. The sign classifies the row:

IRS > 0 · Outperform · #22c55e IRS < 0 · Underperform · #ef4444
Engine D

Day Range Position (%)

LTPt − TLTH − TL × 100

Where the price sits inside today's low→high band. Feeds the Ranking slider filter and the Insights "New Highs/Lows" widget directly; a zero-span guard returns 0 to avoid division errors.

Engine C

Normalized Dual-Range Mapper

x(P) = P − minmax − min × 100

With min = min(YL, TL) and max = max(YH, TH), both sessions map onto one 0–100% scale so yesterday's grey bar, today's blue bar, and the LTP tick share coordinates on a single canvas element.

Engine B (part 1)

30-min Opening Range Breakout (C1–C4)

UpperCn = max(P)  ·  LowerCn = min(P)  over t ∈ Cn

Candles: C1 09:15–09:45 · C2 09:45–10:15 · C3 10:15–10:45 · C4 10:45–11:15. Once a candle completes, a breach of its bound is a candidate signal; the newest completed candle wins, and repeated same-state ticks don't re-fire. (evaluate_orb())

LTP > Upper → Bull • Cn LTP < Lower → Bear • Cn

Engine B (part 2) — the breakout-quality gate

A raw C1 boundary breach isn't shown as a Bull/Bear signal by itself — three additional rules must all hold, gating "Bull • C1" / "Bear • C1" specifically (C2–C4 structural breaks are unaffected). Evaluated live on every tick inside process_incoming_tick().

Rule 1

Opening-range breakout

Price is beyond the 30-min (C1) range — this is Engine B part 1 above. Bull needs LTP > C1 high; Bear needs LTP < C1 low.

Rule 2

First-candle extreme intact

first_candle_extreme_intact() — the very first 5-min candle (09:15–09:20)'s low (bullish) / high (bearish) must still be the day's extreme so far, checked continuously against the live today_low/today_high — not just within the opening 30 minutes, but for the whole day up to now. Fails closed if the candle-1 reference hasn't backfilled yet.

bull: today_low ≥ candle1_low
bear: today_high ≤ candle1_high
Rule 3

Two-sided range

has_two_sided_range() — among the six 5-min candles spanning 09:15–09:45, at least one must close red (close < open) and one green (close > open). Rules out a stock that just ran straight up/down with no two-sided trade at all. Returns False if fewer than six candles are available.

🗓️

Scheduling. The 5-min candles needed for Rules 2–3 don't exist until 09:45 has actually elapsed. A dedicated 09:46 IST cron job (_refresh_opening_range()) re-runs the ORB + quality backfill once real data exists, correcting anything that booted right at market-open before the window had printed.

✓

Verified. 8/8 unit tests pass — python -m tests.test_calculations (pct-change, IRS, day-range, dual-range map, ORB breakout & precedence, two-sided-range, first-candle-extreme-intact).

📊 Ranking 🔥 Heatmap 💡 Insights ⭐ Watchlist

📊 Ranking RankingScreen.jsx

The primary sortable/filterable table (WatchlistRow per stock): Stock · LTP · Price Range (Canvas dual-range bar) · Signal · RS vs NIFTY. Client-side filters (all via useMemo, no re-fetch):

Breakout Signal — All / Bull / Bear, matched against the signal string (which is already server-gated by the 3-rule breakout-quality check above — the filter itself needs no extra logic). Sector — exact match against the raw watchlist sector. Sort — RS ▼/▲, %Change ▼, Day-range % ▼, Symbol A-Z. Day Range Position ≥ N% slider — hides stocks not yet near their daily high.

08 — Heatmap

Heatmap: two-level treemap

HeatmapScreen.jsx + Treemap.jsx. Squarified treemap layout via d3-hierarchy's treemapSquarify tiling — sized rectangles that always stay close to square, unlike a naive slice-and-dice layout. Fully ResizeObserver-responsive.

Level 1 — sector overview (default view)

📐Tile size

Each tile is one NIFTY-style sector group (via sectorGroups.js's niftyGroup() mapping — e.g. both "Energy" and "Power" watchlist sectors fold into NIFTY ENERGY). Size = sum of traded value (ltp × volume) across all members of that group.

🎨Tile color

Color = that group's traded-value-weighted mean %change (sectorAggregates.js's aggregate()) — a heavy-volume stock legitimately dominates the sector's color, unlike the Sector Leaderboard widget which deliberately uses a simple (unweighted) mean instead.

Level 2 — stock drill-down (click any sector tile)

📐Tile size

One tile per stock in the clicked sector group. Size = that stock's individual traded_value.

🎨Tile color

Color = that individual stock's own %change (no aggregation at this level).

Color function (shared by both levels)

function colorFor(pctChange) {
  // capped at ±5% so a couple of extreme movers don't wash out the rest of the map
  const capped = Math.max(-5, Math.min(5, pctChange));
  const intensity = Math.abs(capped) / 5;          // 0..1
  return capped >= 0
    ? `hsl(142, 71%, ${14 + intensity * 28}%)`   // green, darker→brighter
    : `hsl(0, 72%, ${14 + intensity * 26}%)`;    // red,   darker→brighter
}

Tile fills are intentionally theme-invariant (same hsl() regardless of light/dark mode) — a Finviz-style convention where the heat color itself is the whole point of the widget.

09 — Insights

Insights tab: 8 widgets, each a pure derivation of the same stream

Insights.jsx composes all eight below; none of them fetch anything — every calculation is a useMemo over the exact same stocks[] array the Ranking table renders.

📶 Market Breadth MarketBreadth.jsx

Slim single-row banner

Counts stocks with pct_change > 0 (Up) / < 0 (Down) / == 0 (Flat), rendered as one proportionally-segmented bar (green/grey/red) plus raw counts. A one-glance "is today a green day or a red day" gauge.

⚖️ Buy/Sell Pressure BuySellPressure.jsx

Slim single-row banner · needs backend-captured tick fields

Aggregates tot_buy_qty / tot_sell_qty (outstanding order quantities, carried directly on FYERS SymbolUpdate ticks) across the entire watchlist. buyPct = buyQty / (buyQty + sellQty) × 100 (and the complement for sell) — a market-wide order-flow imbalance gauge, not a per-stock one.

🏆 Sector Performance SectorLeaderboard.jsx

Horizontal bar chart, ranked

Ranks every NIFTY sector group by simple (unweighted) mean %change across its member stocks — deliberately not traded-value-weighted, so one heavy-volume stock can't dominate the leaderboard read the way it intentionally does on the Heatmap. Bar width ∝ |mean| / max(|mean|) across all groups; direction (green bar growing right / red bar growing left from center) matches sign.

🚀 Breakout Signals BreakoutLeaderboard.jsx

Live list, sorted by signal time

Every stock whose signal field is currently set (i.e. passed all 3 breakout-quality rules from the Calculations section) — sorted newest-first by signal_time. Bull rows get a green ▲, Bear rows a red ▼.

🔄 Sector Rotation SectorRotationChart.jsx

Hand-rolled scatter chart (no chart library) · absolute-positioned divs

One dot per sector group: x = mean %change today, y = mean relative-strength vs NIFTY. Both axes use a symmetric linear scale (±max(|value|) × 1.15) with the origin at center, splitting the chart into four labeled quadrants:

↖ Improving (RS+, %chg−) ↗ Leading (RS+, %chg+) ↙ Lagging (RS−, %chg−) ↘ Weakening (RS−, %chg+)

Dot diameter scales mildly with member count (10 + min(10, count/2) px) so larger sectors read as visually heavier without dominating the plot.

⚡ Circuit Proximity CircuitProximity.jsx

List, sorted by distance · needs backend-captured tick fields · threshold: 2%

Flags stocks trading within 2% of their exchange circuit limit — a high-signal event in Indian markets. Uses upper_ckt/lower_ckt (also carried directly on FYERS ticks). distPct = (upper_ckt − ltp) / upper_ckt × 100 for the upper side (mirrored for lower); flagged when 0 ≤ distPct ≤ 2. These fields are frequently absent/zero on certain tick types, so this widget shows "no data received yet" more often than the others — that's expected, not a bug.

📈 New Highs / Lows NewHighsLows.jsx

Two side-by-side lists · pure frontend, no new backend field

Derived entirely from day_range_pos (Engine D above): stocks at ≥ 95% of today's range are "near day-high", ≤ 5% are "near day-low". No separate 52-week or all-time high tracking — purely an intraday-range read.

📋 Ranked Movers RankedMovers.jsx

4 mini-lists sharing one layout component, top 8 each

Top Gainers / Top Losers — sorted by pct_change. Most Active (Value) — sorted by traded_value (ltp × volume), formatted in ₹ Cr/L. RS vs Nifty — sorted by relative_strength (Engine A). All four are pure client-side sorts of the same array, no extra backend field required.

10 — Client internals

Frontend rendering

Built for a stable frame rate across 169 live rows, with a full light/dark theme system layered on after the fact without touching per-component logic.

🎨Canvas bars

Pre-normalized coordinates rasterized on <canvas> — no per-pixel DOM, DPR-aware.

🧩React.memo rows

Re-renders isolated to rows/tiles whose data actually changed.

🔎Filters & sort

Signal · sector · day-range slider · multi-key sort, all client-side via useMemo.

📴Offline cache

Each payload mirrored to localStorage; on drop/standby it renders the last snapshot with a Closed/Offline badge.

Light/dark theme system

CSS-variable tokens (tailwind.config.js maps semantic names like surface/primary/accent-blue to rgb(var(--token) / <alpha-value>)) instead of a dark: variant on every className — the codebase already leaned on computed ternary strings, so doubling every one of them would have been far more invasive. ThemeContext.jsx persists the choice to localStorage ("td-theme"); an inline script in index.html applies it synchronously before React mounts to avoid a flash-of-wrong-theme. The one non-CSS exception is OverlappingRangeBar.jsx's Canvas 2D context, which reads the active theme's colors via getComputedStyle() since ctx.fillStyle can't consume Tailwind classes.

11 — Operations

Daily lifecycle

APScheduler on the IST timezone; if the process boots mid-session it jumps straight to the correct state.

☀

08:45 — Daily token

Force-refresh a fresh FYERS access token via the automated TOTP chain (Mon–Fri).

▶

09:15 — Market open

Backfill ranges + ORB + breakout-quality seed, flip market_open = true, launch the websocket thread.

⟲

09:46 — Opening-range refresh

The 9:15–9:45 window has now fully printed; re-run the C1 ORB + breakout-quality backfill so a market-open-time boot (which saw a still-forming candle) picks up real data.

≈

09:15–15:30 — Live

Ticks stream in; SSE broadcasts the computed state once per second.

■

15:30 — Standby

Close the websocket to conserve bandwidth; UI shows Closed with the final snapshot.

🧪

Dev override. FORCE_MARKET_OPEN=true in .env runs the engine regardless of the clock to exercise the full pipeline outside market hours.

12 — Deployment & CI/CD

From push to production

A single Oracle Cloud VM runs everything; GitHub Actions makes every push to main land there automatically.

🌐Caddy

Reverse proxy + automatic HTTPS (Let's Encrypt) for the DuckDNS domain. Routes /api/* and the bare /callback to the backend on localhost:8000; everything else serves the built React SPA with a static-file/try_files fallback to index.html.

⚙️systemd

tradedashboard-backend.service supervises the FastAPI process (Restart=always, unbuffered python -u so logs reach journalctl in real time) — replaced an earlier fragile nohup-in-a-terminal setup that died whenever the SSH session closed.

🚀GitHub Actions

.github/workflows/deploy.yml triggers on every push to main, SSHes in with a repo secret key, and runs deploy/deploy.sh: git reset --hard origin/main → reinstall backend deps → npm ci && npm run build → restart the systemd service → reload Caddy.

🤖

A second, unrelated automation exists. .github/workflows/daily-improvements.yml runs daily at 09:00 UTC, applies black/isort/prettier/safety across the whole repo, and pushes straight to main. Always git pull before pushing new work — a same-day conflict with this bot is almost always pure formatting noise, not a real collision.
